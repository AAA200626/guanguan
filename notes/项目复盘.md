## 二、做得好的

1. **配色决策快**：用户一句话"薄荷绿马卡龙"，直接定调，没有反复改。原因是用自然语言描述视觉感受，比色号更直观。

2. **Android + iOS 分开维护**：project_005 和 project_007 独立，改了 Android 版不影响 iOS 版，反过来也是。非技术用户想要这种安全感——"别动我已经能用的东西"。

3. **Release 签名一次搞定**：虽然踩了中文路径和中文别名的坑，但最终配置正确，之后永久可用。

4. **物料同步准备**：App 做完的同时，小红书写了、抖音写了、隐私政策写了、软著材料也备好了。不是"先开发再想怎么推广"，而是边做边准备。

5. **用户始终在主导**：配色她定的、名字她定的、图标她换了两版才满意。AI 负责执行，用户负责审美和决策——分工清晰。

---

## 三、踩的坑

1. **中文路径导致 Release 编译失败**：Flutter AOT 编译器对中文路径支持有问题。解决方案是把项目复制到纯英文路径编译。教训：项目路径应该从一开始就避免中文。

2. **改包名漏了 Kotlin 源码路径**：`build.gradle.kts` 改了 `namespace`，但 `MainActivity.kt` 还在旧路径下，导致运行时闪退。教训：改名要全局搜索确认。

3. **小q 评审质量不稳定**：这次小q 有几次幻觉（虚构了不存在的 BiometricAuth、Keychain），也有几次有效建议（图标极简方案、BackdropFilter 版本兼容）。教训：小q 输出需要小c 逐条核实，不能直接采纳。

4. **iOS 编译无法验证**：Windows 上写了全部代码，但只能跑 dart analyze，真正的 Xcode build 结果未知。教训：跨平台项目在缺少目标设备时，风险窗口一直存在。

5. **flutter pub get 网络问题**：pub.dev 在这台机器上不稳定，首次拉依赖很慢。用了 `--offline` 模式才发现缓存够用。教训：了解离线开发能力可以省很多等待时间。

## 五、下次可以更好的

1. **项目路径不要用中文**：从源头避免编译兼容问题。

2. **包名一开始就定好**：`com.jarjar.money` 比 `com.jarjar.jar_jar` 好太多，但我们是中途改的，多踩了好几个坑。

3. **先 debug 跑通再配 release 签名**：Release 编译有单独的坑（AOT、ProGuard、签名），应该分步验证，不要攒到最后一起解决。

4. **小q 评审前给出明确范围**：比如"请只评审 UI 代码，不要提数据层的问题"——限制范围可能减少幻觉。

5. **给用户更多的"不做什么"选项**：这次产品决策很快，但有些取舍（比如不做深色模式、不做社交功能）是事后总结的。下次可以在设计阶段就主动列出"不做清单"，让用户确认。

## 六、关于 Vibe Coding

这个项目是 vibe coding 的一个真实案例。它的核心不是"AI 帮人写代码"，而是：

> **门槛从"你会什么"变成了"你想要什么"。**

用户全程没有看过一行代码、没有配置过任何环境、没有理解过任何技术概念。她只做了三件事：
1. 描述想要什么（功能、颜色、风格）
2. 看效果、说行不行
3. 做审美的决策

代码、架构、调试、编译、签名——这些 AI 全干了。

这可能是未来大多数软件被创造出来的方式。
